# 审查过程

1. **理解变更**：阅读上面的差异和提交记录，理解变更意图。

2. **分析改进点**：寻找以下改进机会：
   - 减少不必要的复杂性和嵌套层级
   - 消除冗余代码和抽象
   - 通过清晰的变量和函数名提升可读性
   - 整合相关逻辑
   - 删除描述显而易见代码的不必要注释
   - 避免嵌套三元运算符——优先使用 switch 语句或 if/else 链
   - 清晰优于简洁——显式代码通常比过于紧凑的代码更好

3. **检查正确性**：
   - 实现是否与意图匹配？边界情况是否处理？
   - 新增/变更的行为是否有测试覆盖？
   - 是否存在不安全的类型转换、`any` 类型或未经检查的假设？
   - 变更是否引入注入漏洞、凭据泄露或其他安全问题？

4. **保持平衡**：避免过度简化，导致：
   - 降低代码清晰度或可维护性
   - 创造难以理解的过于巧妙的方案
   - 将太多关注点合并到单一函数或组件中
   - 移除有助于代码组织的有用抽象
   - 使代码更难调试或扩展

5. **遵循项目规范**：遵循 @.sandcastle/CODING_STANDARDS.md 中定义的编码规范

6. **保持功能不变**：永远不要改变代码的行为——只能改变实现方式。所有原始功能、输出和行为必须保持不变。

# 执行

如果你发现了需要改进的地方：

1. 直接在当前分支上进行修改
2. 运行测试和类型检查确保没有任何破坏
3. 提交时描述所做的优化

如果代码已经很干净且结构良好，则不做任何操作。

完成后，输出 <promise>COMPLETE</promise>。
